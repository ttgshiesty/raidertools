# Change-24: Quest Item Tracking

## Status
Implemented

## Summary
Add quest-required item tracking to Quartermaster. Incomplete quests auto-generate lists of items the player must bring on the quest. Pure quest tokens (`questItem: true` in upstream data, e.g. "Moisture Meter") are filtered out ‚Äî only items with broader utility (craftable, consumable, recyclable) are tracked. Quest completion state is determined exclusively from linked ArcTracker/Embark quest snapshots (not manual mode).

## Motivation
The Quests app already tracks quest completion and shows required items per quest in tooltips, but doesn't integrate with inventory planning. Players need to know which items to **bring into a quest mission**, and whether they have enough. By surfacing quest-required items in Quartermaster, the planner can:
- Warn when quest items are missing
- Protect quest-required items from recycling/salvage
- Show quest items in In Raid loot suggestions
- Give per-item provenance showing which quest(s) need each item

## Requirements

### Data Layer (R1)

**R1.1 ‚Äî Propagate `questItem` to generated items**
- Add `questItem?: boolean` to the `SourceItem` interface in `scripts/quartermaster-import.ts`
- Pass it through to the output `PlannerItem` object (read from upstream `source.questItem`)
- Add `questItem?: boolean` to `PlannerItem` and `LocalizedPlannerItemData` in `src/apps/quartermaster/types/item.ts`
- Run `npm run generate:items` after to regenerate

**R1.2 ‚Äî Load quest data in Quartermaster**
- Add `loadQuestData(locale)` to `src/apps/quartermaster/utils/dataLoader.ts`
- Fetches the existing `public/data/quests/quest-data.{locale}.json` (generated by `npm run generate:quests`)
- Returns typed `QuestDefinition[]` with `id`, `name`, `requiredItems: { itemId, quantity }[]`, plus `previousQuestIds` and `nextQuestIds` for hop-distance computation
- All quests are loaded (including those without `requiredItems`), since quests without items may still appear in the dependency chain between item-requiring quests and are needed for BFS traversal

**R1.3 ‚Äî Load linked quest completion**
- Use existing `getCachedLinkedQuestSnapshot()` from `src/shared/services/linkedQuestApi.ts`
- Derive a `Set<string>` of completed quest IDs:
  - For Embark: `questsById[id].state === 'completed'`
  - For ArcTracker: `questsById[id].completed === true`
- If no linked snapshot exists, completed set is empty ‚Üí Quest view shows explainer, no lists generated

**R1.4 ‚Äî Quest definition types**
- New file: `src/apps/quartermaster/types/quest.ts`
```typescript
export interface QuestRequiredItem {
  itemId: string;
  quantity: number;
}
export interface QuestDefinition {
  id: string;
  name: string;
  requiredItems: QuestRequiredItem[];
  previousQuestIds: string[];
  nextQuestIds: string[];
}
```

### List Generation (R3)

**R3.1 ‚Äî `generateQuestLists()`** (new file: `utils/questLists.ts`)
- Generates one `StoredList` per **incomplete** quest that has filtered `requiredItems`
- List ID format: `quest_{questId}`
- List name: the quest's localized name
- List type: `'quest'`
- All quest lists are always enabled (`isEnabled: true`) ‚Äî there is no per-quest enable/disable toggle

**R3.2 ‚Äî Item filtering**
- Exclude items where `itemsMap[itemId]` does not exist (not in items DB)
- Exclude items where `itemsMap[itemId].questItem === true` (pure quest tokens)
- Remaining items each become a `ListItem` with the quest-specified `quantity`

**R3.3 ‚Äî Item toggles**
- First-time generation: all items default to `isEnabled: true`
- Subsequent generations: respect existing per-item toggle state from `questToggles.itemEnabled`
- "Disable All" / "Enable All Pending" tracking modes operate at the item level (not list level)

### Quests View UI (R5)

**R5.1 ‚Äî New top-level view: `QuestsView.tsx`**

- Layout: flex-wrap grid of quest blocks (no accordions / collapsible cards)
- Each quest block: header with quest name (and optional hop-distance badge), then item cards laid out in a wrapping row
- Item cards use the same 84px card layout as the Projects view: icon, item name (2-line clamp), and `got/required` progress text
- Missing items show a red deficit badge in the top-right corner of the icon wrapper

**R5.2 ‚Äî Top controls bar**
- "Sync Quests" button on the left
- "Tracking" label with `qm-segmented-control`: "Disable All" / "Enable All Pending" (operates at item level)
- Sync status label (e.g. "Synced via Embark") ‚Äî source name is capitalized
- Sort switch with `qm-segmented-control`: "A-Z" (alphabetical) / "Next Quests" (with hop-distance icon)
- Sort preference is persisted to `localStorage` (`quartermaster.ui.questSortMode`) and survives view switches

**R5.3 ‚Äî Sorting algorithms**
- **Alphabetical (A-Z)**: quests sorted by localized name (default)
- **Next Quests**: BFS hop distance from frontier quests (available or active). Available quests show "Now" badge (hop 0); subsequent quests show numeric hop count. Unreachable quests sort last. Same-hop quests are sub-sorted alphabetically.

**R5.4 ‚Äî No-data state**
- If no linked quest snapshot exists: show explainer panel with message

- If linked but no incomplete quests have required items: show empty state

**R5.5 ‚Äî Badge counts**
- Sidebar badge: number of quests where all enabled items are fully owned ("ready to go")

## Edge Cases

| Scenario | Behavior |
|----------|----------|
| No linked quest snapshot | Quests view shows explainer message; no quest lists generated; planner unaffected |
| Snapshot exists but no incomplete quests | Empty state in Quests view |
| Quest has requiredItems but all are `questItem: true` | Quest is skipped (filtered out entirely) |
| Quest has requiredItems but some items are not in items DB | Those items are silently dropped; remaining items generate list |
| User completes a quest via sync | List disappears on next generation (quest now in `completedQuestIds`) |
| Quest ID in snapshot not in quest data | Silently ignored (no list generated) |
| Stale linked snapshot | Use last cached data; "Sync Quests" button available to refresh |
| Intermediate quest in chain has no requiredItems | BFS still traverses through it (all quests loaded regardless of item presence); hop distance computed correctly |
| Quests with unreachable dependencies | Unlimited hop distance; sorted last in "Next Quests" mode |
| Null hop distance | Quest rendered without hop badge |
